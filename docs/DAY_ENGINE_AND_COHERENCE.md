# Day Engine и цельность продукта (аудит → направление)

**Статус:** канон продуктово-инженерного направления (не маркетинг).  
**Связь:** [PERSONAL_INTELLIGENCE_LAYER.md](./PERSONAL_INTELLIGENCE_LAYER.md), [TODAY_PERSONALIZATION_CORE.md](./TODAY_PERSONALIZATION_CORE.md), [CORE_PRODUCT_CANON.md](./CORE_PRODUCT_CANON.md). **Выполнение:** чеклист DE-1…DE-13 и порядок работ — [PRODUCT_EXECUTION_TRACKER.md](./PRODUCT_EXECUTION_TRACKER.md) §4.7, §5.3.  
**Паритет:** web и iOS должны опираться на одни и те же контракты данных и один смысл «дня».  
**Центральная логика:** [§10 DayModel](#10-daymodel-и-decision-engine-закрытый-канон) — без этого слоя текст и UI не имеют опоры.

---

## 1. Корневая проблема

Сейчас система **генерирует и показывает куски**, но не всегда собирает их в **одну причинно-следственную модель дня** для пользователя и для модели.

Следствие:

- тексты, карта, число, действия и UI **слабо объясняют общую логику** («почему всё вместе»);
- для обучения и для доверия не хватает явной цепочки: **причина → интерпретация → рекомендация**.

**Цель:** ввести единый **Day Engine** — **Decision Engine for the day**: на входе **DayContext** (факты и пользователь), в середине обязательный **DayModel** (логический объект дня, не лонгрид), на выходе **решение + объяснение** (тексты и UI только проекции DayModel). События и фидбек замыкают цикл.

## 2. DayContext (единый источник правды дня)

Целевой **логический** объект (в коде может быть структура в бэке + сериализация в промпт; не обязательно один физический JSON в БД с первого дня):

| Слой | Содержание | Уже близко в коде / данных |
|------|------------|---------------------------|
| `tarot` | Карта дня (ритуал), применение к дню | `ritual_context`, персист ритуала |
| `numerology` | Число дня | morning + ritual |
| `astrology` | Стержень дня, риск, режим, луна/транзиты в доступном виде | `daily_foundation` / morning / `why_astrological_layers` в narrative |
| `user_profile` | Профиль, натал при наличии | `core_profile` / `user_core` в narrative |
| `mood` | Состояние после ритуала | ritual + tracking |
| `behavior_patterns` | Агрегаты «часто / избегает / перегружает» | learning-слой, `dominant_diary_topics` — задел |
| `history` | Вчера, неделя, повторяющиеся темы | **пробел:** слабая явная подача в один пакет |
| `intent` | «Что важно сегодня» (цель дня, тема головы, ответы мини-вопросов) | **DE-6:** `layers.intent` (`intent_slice_v0`): `DayConnection` + `head_topic` из `ritual_context` → промпты narrative (`today-narrative-v9`), кэш по `intent_context_fp` |
| `visible_profile` / `internal_profile` | Разделение UX-фактов и агрегатов для LLM | **DE-12:** `profile_prompt_slices_v0` → `DayContext.layers` + user JSON всех narrative surface; системный блок в промпте v9 |
| `rhythm` | Цели, привычки, аскезы, отметки, **сводка дневника** | `fusion.rhythm_context` в промпте narrative; дневник как факт, не выдумка |
| `health_signals` | Сон, активность (с согласия пользователя) | **пробел:** интеграция Health |

**Правило Day Engine (вход):** в LLM для поверхностей Today уходит **один согласованный пакет входа** (DayContext + версии), а не разрозненные догадки по отдельным полям.

**Правило Day Engine (выход — целевое):** не пытаться получить **одним** широким запросом весь смысл дня и все блоки UI сразу: это вынуждает модель усреднять. Цель — **управляемая цепочка вывода** (узкая задача → ограниченный JSON → следующий шаг опирается на `parent_generation_id` / артефакты предыдущих логов). Текущий `surface=guide` — монолитный компромисс; декомпозиция — см. **§2.1** и **DE-13** в [PRODUCT_EXECUTION_TRACKER.md](./PRODUCT_EXECUTION_TRACKER.md) §4.7.

### 2.1 Цепочка смыслов (не один промпт на всё)

Целевая логика: **факты → интерпретация → персонализация → блоки интерфейса → действия → обратная связь**. Каждый этап — отдельный узкий вызов к модели (или детерминированный слой без LLM), с явным входом и ограниченным выходом; пропуск уровня даёт «красивый мусор».

| Уровень | Роль | Что на выходе | Приближение в коде сегодня |
|---------|------|----------------|----------------------------|
| **1** | База дня (объективно) | Сухая структура: напряжение, окно, тип действия, риск — **не** пользовательский лонгрид | `daily_foundation`, расчёты утра, детерминированный `day_engine_brief` |
| **2** | Интерпретация | Связка карта + число + астрология в одну картину дня — **объяснение**, не советы | частично смешано с guide; **цель** — отдельный артефакт/вызов |
| **3** | Персонализация | Результат уровня 2 + внутренний профиль / паттерны → корректировка тона и реалистичности шага | `internal_profile`, `behavior_patterns`, `user_core` в одном пакете; **цель** — отдельный вызов после фиксации уровня 2 |
| **4** | Блоки UI | Сводка, сферы, вектор и т.д. — **по отдельности**, общий DayContext + ссылки на предыдущие поколения | сейчас один JSON guide; **цель** — несколько узких surface или подшагов с `parent_generation_id` |
| **5** | Действия | 2–4 **выполнимых** шага, согласованных с днём и паттернами | `action_options` в guide; **цель** — отдельный узкий вызов |
| **6** | Реакция пользователя | Выбор / игнор / сделано → события → следующий цикл | meaning events, `generation_id`, вечер (DE-3–DE-5) |

**Приоритет починки (продукт):** сначала убрать текстовый шум и выровнять один якорный блок «Твой день»; затем **внутри бэкенда** ввести pipeline уровней 1→3 (факты и интерпретация можно усилить детерминированно до разнесения LLM); потом разнести блоки (4–5); UI не впереди цепочки смысла.

**Спецификация и схема v0:** [DAY_CONTEXT_V0.md](./DAY_CONTEXT_V0.md) · `docs/schemas/day_context_v0.schema.json` · сборщик-черновик `build_day_context_v0` в бэкенде (`todayflow_backend.services.day_context`).

**Входные шкалы Reference → DayModel:** [DAYMODEL_INPUT_CONTRACT.md](./DAYMODEL_INPUT_CONTRACT.md) (P0.1 — до миграции JSON справочников).

**Дневник и LLM (уже в контуре):** в системном промпте narrative указано использование `fusion.rhythm_context` (цели, привычки, аскезы, **сводка дневника** за окно). В learning-контексте для narrative участвуют и темы дневника (`dominant_diary_topics` в сборке user-пакета). Расширение: явные **короткие выдержки** (с лимитом и политикой приватности) только при согласии и версии контракта.

**Здоровье (Apple Health / HealthKit; позже Android Health Connect):** целевой вклад в DayContext — **сон и активность** как факты ритма (не диагноз): улучшить объяснимость «ресурса дня» и паттерны «утро слабее / перегруз». Требуется: явное согласие, минимизация данных, отдельный ingestion и поля в DayContext, паритет описания в iOS и на вебе (где применимо).

## 3. Объяснимость (Explain layer)

Каждый заметный вывод для пользователя должен быть **прослеживаем**:

1. **Причина** (факт: карта, число, слой дня, настроение, паттерн, вчерашний хвост).  
2. **Интерпретация** (коротко, без приговора).  
3. **Рекомендация** (конкретное действие или «чего не делать»).

В продукте: блок **«Почему так?»** и связка карта/число/астро с текстом дня — must-have; дальше — структурированные поля в ответе API (см. `why_astrological_layers`, `TODAY_PERSONALIZATION_CORE` для эволюции контракта). Конкретика по полям `core_message` / `action_options` для `surface=guide` — **§9**.

## 7. Порядок работ (рекомендуемый)

1. **Зафиксировать и внедрить DayModel** (§10): логическая схема → артефакт в бэкенде → валидация «нет Vector/Tension/Risk — нет пользовательского текста». Без этого шага не расширять UI.  
2. Убрать мусор из UI и дубли; один якорный блок «Твой день», опирающийся на DayModel, как только он появился в данных.  
3. **DayContext** (схема, сборщик, одна точка **входа**) — DE-1…DE-2; питает нормализацию фактов для DayModel.  
4. Развести **генерацию по §2.1 и §10.5** (узкие вызовы / DE-13), а не один монолитный промпт на всё.  
5. Переписать/усилить каждый шаг под **причинно-следственную** подачу (промпт + JSON + валидация).  
6. Замкнуть **feedback loop** на шаги и вечер.  
7. Накопление поведенческих сигналов и паттернов.  
8. Затем — расширение модели / fine-tuning на накопленном.

Параллельно в бэклоге: **Health** (сон/активность), углубление **дневника** в DayContext по политике приватности.

## 8. Итог одной фразой

Продукт становится цельным, когда **DayContext** питает **Day Engine**, внутри которого зафиксирован **DayModel** (§10), а **события и фидбек** замыкают цикл «решение → действие → исход».

## 9. Контракт `surface=guide` (паритет web ↔ iOS, 2026)

Один ответ `POST /today/narrative` с `surface=guide` задаёт «слой объяснимости» после ритуала. Чтобы веб и iOS не расходились:

| Поле | Форма | Смысл |
|------|--------|--------|
| `core_message` | **строка** (legacy) **или объект** | Цепочка «смысл → риск → ход»: объект с полями `body` (обязательно), опционально `headline` / `title`, `risk` / `main_risk`, `best_move` / `first_move`. Строка — 2–4 абзаца через `\n\n`. |
| `action_options` | массив из **3** элементов: **строка** или **объект** | Объект: `title` (обяз.), опционально `reason` / `why`, `estimated_minutes`, `entity_kind` / `creates`. Клиенты без rich-UI показывают только `title`. |
| `sphere_triad`, `support_hooks`, `why_astrological_layers` | без изменений по роли | Причинность и «Почему так» остаются в слоях и триаде. |

**Бэкенд:** `today_narrative._validate_payload_shape` / `_ensure_guide_actionable_fields` принимают и нормализуют обе формы; промпт `_GUIDE_SYS` описывает объектный вариант как предпочтительный для причинности.

**Веб:** `todayGuideActionable.ts` — `parseCoreMessageForUi`, `parseActionOptionsRich`.

**iOS:** `TodayGuideActionable.swift` — `coreMessageDisplayString`, `coreMessageParagraphs`, `parseActionOptions` (те же ключи, что на вебе).

Дальнейшие поля (`feedback_questions`, `evening_prompts` в одном JSON) — отдельная итерация поверх этого слоя, без ломки обязательных ключей выше.

**Эволюция:** §9 фиксирует текущий **монолитный** ответ guide для паритета web/iOS. Целевая архитектура — **несколько узких вызовов** по цепочке §2.1 и артефакту **DayModel** (§10.5) с теми же контрактами на уровне блоков или с версией контракта v2; см. **DE-13** в трекере §4.7.

## 10. DayModel и Decision Engine (закрытый канон)

### 10.1 Что строится

Не «астро-приложение» и не «генератор текстов». Строится **Decision Engine for the day**: система, которая понимает контекст дня, понимает человека, сводит это в **конфликт**, даёт **реалистичное решение**. Продуктовый минимум дня — не прогноз, а **решение + объяснение**.

### 10.2 DayModel — один логический объект (не текст)

Всё крутится вокруг **DayModel**: структура из **шести обязательных параметров**. Если любой из них не определён осмысленно — пользовательский текст по дню **не генерируется** (только доработка модели/входа).

| Параметр | Смысл |
|----------|--------|
| **Vector** | Куда «толкает» день (доминирующее направление). |
| **Tension** | Где конфликт **между источниками** (карта / число / астрология не сходятся в одну линию). |
| **Opportunity** | Где Vector можно реализовать (окно). |
| **Risk** | Где человек **сломается** (Tension + поведение и состояние пользователя). |
| **Tempo** | Как двигаться (согласованная скорость дня по энергии/сигналам). |
| **Strategy** | Что делать, чтобы день сработал: соответствует Vector, снижает Risk, учитывает пользователя. |

Связь с пятью «смысловыми столпами» дня: Vector ≈ базовый вектор; Tension ≈ зона напряжения; Opportunity ≈ окно; Risk ≈ тип ошибок; Strategy ≈ оптимальная стратегия; **Tempo** выведен явно как шестой параметр (не смешивать со Strategy).

### 10.3 Жёсткая логика построения дня

**Шаг 1 — Факты (сырой слой).** Собираются астрология, карта, число (и при необходимости луна/стержень из foundation). Это **сырьё**, не пользовательский вывод.

**Шаг 2 — Сведение (ядро).** Агрегация в **одну картину**: не три отдельных «мнения», а **единый смысл** дня. Правило агрегатора: если **два и более** источника по нормализованным шкалам совпадают → это **доминанта** (в т.ч. задаёт Vector); если источники **конфликтуют** → это **Tension**. На выходе шага 2 — заполненный каркас DayModel по полям, которые можно вывести из фактов без пользователя (часть Risk до пользователя — заглушка или «низкая уверенность» до шага 3).

**Шаг 3 — Пользователь.** Накладываются паттерны, поведение, текущее состояние (в т.ч. `internal_profile`, intent, ритуал). Главный вопрос: **где конфликт между днём и человеком?** Усиливается **Risk**, при необходимости меняется **Strategy** (не общие слова, а сдвиг стратегии: например день требует спокойного завершения, пользователь ускоряется → Strategy = ограничение до одного шага, а не «будь осознанным»).

**Шаг 4 — Решение.** Из конфликта выводится смысл для пользователя: если действовать по привычке → один исход; если скорректировать по Strategy → другой. Только здесь допускается переход к формулировкам «что происходит / почему».

### 10.4 Нормализация входа (обязательная)

Все источники переводятся в **один язык шкал** (фиксированный набор, без произвольных синонимов в промпте):

- **Направление (Vector-ось):** завершение / рост / стабилизация / конфликт / переход.  
- **Энергия:** низкая / средняя / высокая.  
- **Эмоции:** стабильные / чувствительные / искажённые.  
- **Тип действия:** начать / продолжить / закончить / не трогать.  
- **Темп:** медленно / ровно / быстро.

Каждый источник (карта, число, астрология) выставляет значения по этим шкалам; агрегатор на шаге 2 работает **только** с этими значениями, не с «красивым текстом».

### 10.5 Формула (ядро)

- **Vector** = доминирующее направление дня после сведения источников.  
- **Tension** = конфликт между источниками (несовпадение шкал).  
- **Opportunity** = зона, где Vector реализуем.  
- **Risk** = Tension, уточнённый поведением и состоянием пользователя (шаг 3).  
- **Tempo** = согласованный темп (из энергии/сигналов источников и при необходимости коррекции под пользователя).  
- **Strategy** = действие (класс действия), которое: согласовано с Vector, снижает Risk, учитывает пользователя.

### 10.6 Минимальный результат для пользователя

Любой допустимый дневной вывод должен закрывать пять вопросов читателя:

1. Что сегодня происходит.  
2. Почему.  
3. Где я облажаюсь (риск).  
4. Что реально сработает.  
5. **Один** фокус.

Если любой пункт не читается из выдачи — результат **бракуется** и пересобирается на уровне DayModel/промпта, а не «косметикой» предложений.

### 10.7 Как из DayModel появляется текст

Текст **не** генерируется «с нуля». Сначала фиксируется **DayModel** (структура), затем LLM **разворачивает** его в прозу в рамках полей API (сводка, do/avoid, фокус). Порядок: **DayModel → текст**.

### 10.8 Сферы и шаги (не из воздуха)

- **Сферы:** проекция DayModel на оси сфер (пример логики: Vector = завершение → приоритет дел/работы; нестабильные эмоции → осторожность в отношениях; низкая энергия → не разгонять деньги/крупные решения). Правила проекции задаются явно в продукте/бэкенде, а не выдумываются моделью.  
- **Главный шаг / action_options:** не придумывается отдельно от **Strategy**; из Strategy выводятся **2–3 конкретных** варианта одного класса (например «один шаг без перегруза» → закрыть задачу / закончить разговор / убрать хвост).

### 10.9 Несколько запросов к модели (правильная схема)

Один широкий запрос на всё запрещён как целевое состояние. Разнесение:

| Запрос | Задача | Вход | Выход |
|--------|--------|------|--------|
| **1** | Сбор **DayModel** из фактов | карта, число, астрология (нормализованные шкалы) | Vector, Tension, Opportunity, Tempo + черновик Risk без пользователя |
| **2** | Адаптация под пользователя | DayModel + профиль (внутренний/видимый) | Risk, Strategy (конфликт день↔человек) |
| **3** | Ядро текста | DayModel + политика голоса | сводка, do, avoid, один фокус |
| **4** | Блоки | тот же контекст + `parent_generation_id` | сферы, поддержка, уточнения |

Точная нарезка по эндпоинтам и кэшу — **DE-13**; логика таблицы неизменна.

### 10.10 Проверка на качество (гейт)

Перед показом пользователю команда должна ответить **три** вопроса по заполненному DayModel:

1. Какой сегодня **Vector**?  
2. Где **Tension**?  
3. Какой **Risk** (с учётом пользователя)?

Если на любой вопрос нет честного ответа из данных — **генерации пользовательского текста дня нет** (fallback: уточнить вход, обеднить UI, или повторить узкий вызов с тем же DayContext).

### 10.11 Почему раньше ломалось

Не было **единого слоя смысла (DayModel)** и явного **конфликта**; сразу шла генерация текста → LLM усреднял и «красиво ни о чём». Исправление — не больше итераций копирайта, а **внедрение DayModel и гейта §10.10**.

### 10.12 Реализация v0 (факт в коде)

- **Бэкенд:** детерминированный **`day_model_v0`** (`build_day_model_v0`) в **`DayContext.layers.day_model`**; в ответе **guide** тот же объект в **`payload.day_model`** (в т.ч. при cache hit — пересчёт с текущими входами). Контракт JSON: `docs/schemas/day_context_v0.schema.json`, фикстура `docs/schemas/fixtures/day_context_v0.valid.json`.
- **Narrative:** системный промпт согласует текст с полями `day_model`; в логе поколений при наличии слоя — **`day_model_contract`: `day_model_v0`** (все surface). В user JSON для **day_layer**, **spheres**, **evening**, **deepen** передаются **`day_model`** и **`day_engine_brief`** (как у guide), чтобы дочерние тексты не расходились с главным смыслом дня. Версия промпта модуля: **`today-narrative-v11`**.
- **Веб / iOS:** в **`TodayRitualFlow`** / **`TodayRitualFlowView`** и на главной вкладке Guide (**`TodayGuideSection`** / **`TodayGuideNarrativeHero`**) при наличии данных — блок «Опора дня» и при `day_model` строки **вектор** (`vector.summary`) и **фокус** (`strategy.one_focus`); если `day_engine_brief` нет — отдельный блок «Логика дня».
