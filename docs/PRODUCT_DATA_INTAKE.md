# Product Data Intake — ровно два способа

**Статус:** TARGET DRAFT — продуктовая фиксация  
**Роль:** единственный канон **как данные попадают в систему**. Не пайплайн, не UI-рефактор.  
**Связь:** [PRODUCT_AVAILABILITY_MATRIX.md](./PRODUCT_AVAILABILITY_MATRIX.md) (слой 0) · [UNDERSTANDING_PROGRESS_AND_DEPTH_CANON.md](./UNDERSTANDING_PROGRESS_AND_DEPTH_CANON.md) · Compat content  
**Правило:** отдельных анкет для Profile / Compatibility / Natal / Today **нет**. Все поверхности читают **сохранённые профили**.

---

## Итог в одном абзаце

Способов ввода данных для построения профиля — **ровно два**.

1. **Новый пользователь без аккаунта** — публичный продукт (совместимость **или** «построить мой профиль») → создаются полноценные профили → ограниченный preview → email → подтверждение → аккаунт → trial/full.  
2. **Авторизованный пользователь** — «Добавить профиль» → те же поля → сохранение в аккаунт **без** email.

Всё остальное — вариации внутри этих двух, не отдельные сценарии.

## Сценарий 1 — Новый пользователь без аккаунта

Пользователь на публичной части. Две **точки входа**, один механизм сохранения.

### 1A. Совместимость

**Сначала — игровая совместимость (без регистрации):**

- совместимость знаков;
- разные жизненные ситуации;
- лёгкий развлекательный формат.

**Рядом — глубокий продукт:** «Посмотреть персональную совместимость двух людей.»

Для персональной совместимости вводятся данные **двух сторон**:

| Поле | Обязательность |
|------|----------------|
| Дата рождения | **обязательна** для базового разбора стороны |
| Имя | опционально (слой именной нумерологии) |
| Место рождения | нужно вместе со временем для ASC/домов |
| Время рождения | опционально; без него — нет ASC/домов/углов |

Частичный ввод **допустим**. Система **не** блокирует весь разбор — доступность каждого вывода по [Availability Matrix · слой 1](./PRODUCT_AVAILABILITY_MATRIX.md).

**Критично:** каждая сторона сразу создаётся как **отдельный профиль** (Profile A, Profile B) — не временные поля формы и не данные «внутри» объекта совместимости.

Совместимость ссылается на:

- `profile_a_id`
- `profile_b_id`

#### До регистрации (preview)

**Доступно:** общее описание связи · точки притяжения · различия · базовая совместимость по доступным данным · объяснение ограничений из‑за missing fields.

**Недоступно:** сексуальная совместимость · глубокие инсайты · конфликтные сценарии · «что делать» · практики улучшения · чувствительные/детальные разделы · история и сохранённый доступ.

#### Сохранение через email

1. Пользователь вводит email.  
2. Система **сохраняет оба профиля** и рассчитанную совместимость.  
3. Письмо со **ссылкой подтверждения** email.  
4. После подтверждения — аккаунт.  
5. **Безопасность (TARGET):** не слать готовый пароль по email → «Email подтверждён. Установите пароль.» (magic / set-password link).  
6. Вход → сохранённые профили и разбор.

Регистрация **не создаёт профили заново** — только закрепляет уже созданные за `account_id`.

#### Trial

После подтверждения: либо явное согласие на trial, либо автозапуск с **заранее** ясными условиями.  
Молча начинать платный / автопродлеваемый trial — **запрещено**.

После старта trial открываются закрытые части preview (секс · глубокие инсайты · конфликты · рекомендации · полный разбор · история).

---

### 1B. Построить мой профиль / карту

Тот же процесс для **одного** профиля (владельца):

1. имя · дата · место · время (частично OK);  
2. создаётся профиль;  
3. доступный разбор + явные ограничения;  
4. сохранить через email;  
5. подтверждение → аккаунт;  
6. профиль уже внутри аккаунта.

#### До регистрации

**Доступно:** базовая часть профиля · доступные расчёты · короткое объяснение ключевых характеристик · список недоступного без времени/места/имени.

**Недоступно:** полный глубокий разбор · часть инсайтов · практические рекомендации · расширенные сферы · история · полная персонализация Today · premium-разделы.

**Главное:** регистрация только **привязывает** профиль к аккаунту.

## Сценарий 2 — Авторизованный: добавить профиль

Единственный другой способ.

CTA: **Добавить профиль**

Те же поля (частично OK). Email **не** спрашиваем. Новый аккаунт **не** создаём.

Система:

1. создаёт самостоятельный профиль;  
2. сохраняет введённые данные в нём;  
3. считает доступные производные;  
4. отмечает недоступные разделы;  
5. профиль доступен в совместимости, карте, Profile и др.

### Метка отношения (не замена профиля)

`relationship_to_owner`: `self` · `partner` · `child` · `relative` · `friend` · `other`  
+ `is_owner_profile` для профиля владельца аккаунта.

---

## Единая модель данных

Все введённые данные живут в **профиле человека**, не в форме, не в сессии как SoT, не как копия birth data внутри совместимости.

### Минимальные поля профиля (TARGET)

| Поле | Смысл |
|------|--------|
| `profile_id` | id |
| `account_id` | null до привязки к аккаунту; после email-confirm — владелец |
| `display_name` | имя |
| `birth_date` | дата |
| `birth_time` | время |
| `birth_time_known` | известность времени |
| `birth_place` | исходный ввод места |
| `latitude` / `longitude` | координаты |
| `timezone_at_birth` | IANA / resolved TZ |
| `relationship_to_owner` | метка связи |
| `is_owner_profile` | профиль владельца |
| `data_completeness` | сводка доступности слоёв |
| `created_at` / `updated_at` | |

### Дополнительно

- нормализованное место;  
- источник координат;  
- статус точности времени;  
- какие расчётные возможности доступны;  
- какие данные отсутствуют.

### Три слоя хранения (не смешивать)

| Слой | Содержание |
|------|------------|
| **Введённые данные** | имя · дата · время · место (как сказал человек) |
| **Рассчитанные данные** | планеты · знаки · аспекты · ASC · дома · нумерология · флаги доступности |
| **Сгенерированные разборы** | персональный профиль · совместимость A↔B · Today · другие чтения |

Совместимость = результат с `profile_a_id` + `profile_b_id`.  
**Не** копировать birth data пары внутрь совместимости как основной источник.

## Схема потоков

```text
Новый пользователь
  Публичный продукт (1A или 1B)
    → ввод данных
    → создание 1 или 2 профилей (durable)
    → расчёт
    → ограниченный preview
    → email
    → подтверждение
    → аккаунт (привязка профилей)
    → trial / full access

Авторизованный
  Добавить профиль
    → ввод данных
    → создание профиля
    → расчёт
    → использование в продукте
```
