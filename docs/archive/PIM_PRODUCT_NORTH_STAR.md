## 0. Диагноз (почему спорили)

Большинство прошлых споров про **тексты · экраны · карточки · порядок · прогнозы · чек-ины** — симптом одной проблемы:

> **Не было критерия успеха продукта.**

Без него команда оптимизировала **Experience**, не **актив платформы**.

### 0.1 Параллель HostFlow

| | HostFlow (раньше) | TodayFlow (сейчас) |
|---|-------------------|---------------------|
| **Как воспринимали** | CRM | приложение про карты · прогнозы · экраны |
| **Споры** | таблицы · формы · карточки | тексты · блоки · ритуал · чек-ины |
| **Перелом** | «Что **источник истины**?» | **PIM** — источник истины **о человеке** |
| **После** | операционная платформа вокруг SoT | Experiences = потребители; Learning Δ |

Пока центра не было — споры на уровне UI бесконечны. После появления центра многие решения **перестают требовать отдельного UI-обсуждения** — они следуют из цели PIM (§2).

### 0.2 Три слоя — не путать *(2026-06-23 · AR-012)*

| Слой | North Star / роль | Вопрос |
|------|-------------------|--------|
| **Experience** | **Ежедневная полезность** — не retention, не data | Становится ли жизнь **ощутимо лучше** от завершённого дня? |
| **Поведение** | **Retention** — *индикатор*, не цель | D1 · D7 · завершённые дни · ≥5 return / 14d |
| **Platform** | **Learning Δ / PIM** — инфраструктура | Завершённый цикл увеличил ценность модели? |

**Жёсткая причинность** *(каждый следующий — следствие предыдущего)*:

```
Daily usefulness → Retention → Data
```

Обратное **не** работает:

```
Data ↛ Retention ↛ Usefulness
```

Можно собрать данные без продукта · можно создать **липкий ритуал без ценности** (streak без результата). Duolingo держат **язык**, не streak; Strava — **прогресс**, не открытие app.

**Главный вопрос TodayFlow:**

> **Что пользователь получает вечером такого, чего не получил бы без TodayFlow?**

Слабый ответ → D7/D30 не спасут. Сильный ответ → retention **следствие**.

**Продуктовая ценность** *(что должно появиться у человека)*:

- больше **ясности**;
- больше **выполненных важных вещей**;
- меньше **сожалений**;
- лучшее **понимание себя**;
- более **осознанный следующий день**.

**Daily value test:** убрать все будущие исследования — остаётся ли **самостоятельная ежедневная ценность**?

**Evening screen test (S0–S10):** увеличивает ли экран вероятность, что вечером пользователь скажет *«этот день был лучше благодаря TodayFlow»*? Если экран отвечает только «какие данные соберём» → инфраструктура.

**Launch one-liner:** не сбор данных · не streak ради streak — **день, который вечером ощущается полезнее**. Тогда PR2 · IR · PIM получают шанс.

**AR freeze *(2026-06-23)*:** новые AR **не добавлять**. Ответ на *какое **поведение** меняется* — [BEHAVIOR_CHANGE_TEST_V0.md](../status/BEHAVIOR_CHANGE_TEST_V0.md).

## 1. Критерий успеха платформы (одна строка)

> **Завершённый день должен увеличивать ценность PIM.**

**Применимо**, когда день дал **ощутимую пользу** (AR-012). Без полезности Learning Δ = шум; retention без пользы = липкий ритуал.

Не путать с Experience North Star (§0.2): там — **ежедневная полезность**; retention — **индикатор**; здесь — **прирост модели за цикл**.

## 2. North Star платформы

| Продукт | North Star (формулировка) |
|---------|---------------------------|
| **HostFlow** (исторически) | Источник истины → потребители |
| **TodayFlow** (формируется) | Каждый **законченный цикл** взаимодействия должен **помочь пользователю** и/или **уменьшить неопределённость модели человека**. Желательно — **и то, и другое**. |

**Операционно:** `Learning Δ > 0` за цикл → платформа выросла. `Learning Δ = 0` → продукт не стал лучше, даже при красивом UI.

### 2.1 Решения, которые следуют из PIM (не спорят на UI)

| Решение | Потому что цель PIM — … |
|---------|-------------------------|
| Discovery ≠ mood picker | улучшать модель (LRE, uncertainty) |
| Goal Loop ≠ только guidance | улучшать модель (Intent, evidence) |
| trait ≠ один ответ | доказуемое знание (C14) |
| Contradiction Event | оставаться **правдивой** моделью (C15–C16) |
| Decision Relevance | помогать **принимать решения** (C17) |

## 3. Два контрпримера

### Хороший экран, плохой продукт

Пользователь: карта → число → красивый синтез → закрыл приложение.

| | |
|---|---|
| Experience | ✓ приятно |
| PIM | **не изменился** |
| Learning Δ | **0** |
| Вердикт | продукт **не** улучшился |

### Средний экран, хороший продукт

Пользователь: цель → частично выполнил → один discovery → contradiction по гипотезе.

| | |
|---|---|
| Experience | ~ нормально |
| PIM | **богаче** |
| Learning Δ | **> 0** |
| Вердикт | **платформа выросла** |

## 4. Приоритизация (ближайшие месяцы)

**Главный вопрос — не:**

> Какой следующий экран?

**А:**

> **Какие взаимодействия дают максимальный прирост PIM на единицу сложности реализации?**

Это ближе к собственной модели, чем очередной UI-компонент или правка текста.

## 5. PIM ROI (проверка для PR и roadmap)

Для **нового сценария** — три вопроса:

| Вопрос | Пример |
|--------|--------|
| **Что получает пользователь?** | guidance, focus, insight |
| **Что получает PIM?** | signal, evidence, contradiction, atom candidate |
| **Стоит ли знание стоимости реализации?** | да / нет — явно в PR |

### Высокий Experience, низкий PIM ROI

Experience лучше · PIM нет · стратегическая ценность **низкая**:

- красивые анимации карт;
- ещё одна колода;
- ещё один тип числа;
- ещё один гороскопный блок.

### Низкий «блеск», высокий PIM ROI

Может выглядеть скучно · сильно улучшает модель:

- Intent Record + outcome;
- post-goal behavior;
- targeted discovery на `hypothesis_id`;
- contradiction с `change_nature`.

**PR template:** секция `## PIM ROI` — [PIM_PR_GATE_V1.md](../pim/PIM_PR_GATE_V1.md) §1.6.
