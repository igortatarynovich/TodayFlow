## Главная норма

> **TodayFlow не генерирует смысл из пустоты.**  
> Система сначала рассчитывает и собирает подтверждённые сигналы, затем выводит структурированное значение, связывает его с реальным контекстом пользователя и только после этого формирует понятный практический текст.

Ещё жёстче:

> **Если команда не может объяснить происхождение, расчёт и практический смысл вывода — этот вывод не должен попадать в production.**

Это обязательное правило для каждого endpoint, каждого промпта, каждого контракта и каждого UI-блока.

## Универсальная цепочка

Любой пользовательский вывод — результат понятной **внутренней** цепочки:

```text
источник → расчёт → интерпретация → практический смысл → финальный текст
```

| Шаг | Кто делает | Что нельзя |
|-----|------------|------------|
| **1. Источники** | сбор из данных пользователя / Reference / истории | выдумывать отсутствующие факты |
| **2. Расчёт** | **детерминированный код** | «угадывать» асцендент, числа, дома моделью |
| **3. Интерпретация** | правила · веса · конфликты · словарь | свободная эзотерическая фантазия |
| **4. Практический смысл** | привязка к реальному контексту пользователя | абстрактные сферы и мотивационные ярлыки |
| **5. Финальный текст** | LLM / шаблон **только поверх структуры** | красивая бессмыслица без шагов 1–4 |

Пользователь видит только шаг 5 (и мягкое «почему» по смыслу).  
Кухня шагов 1–4 обязана существовать, храниться и быть проверяемой.

Связь с [EXPLAIN_MEANING_NOT_MECHANISM.md](./EXPLAIN_MEANING_NOT_MECHANISM.md): кухня **не** показывается как «вас анализирует алгоритм»; пользователю — грамотный полезный вывод.

---

## Область действия (всё продукт)

Цепочка обязательна для:

- целей на день / месяц / год  
- рекомендаций  
- обещаний и аскез  
- практик  
- карты дня и числа дня  
- натальной карты (в т.ч. асцендент, дома)  
- совместимости  
- профиля и итоговых текстов  
- DailyState / guidance / Tarot / Numerology  
- **любых будущих разделов**

Нет исключений вида «это же просто красивый CTA».

## Главный запрет: осмысленно звучащая пустота

Недействительны (и не публикуются), пока нет полной цепочки:

- «один важный разговор»  
- «одно дело до конца»  
- «устойчивость через ритм»  
- «мягко проявить себя»  
- «удерживать внутреннюю опору»  
- «выбрать главное»  
- «довериться потоку»  
- «избегать: семья / можно: работа»

Такие фразы допустимы **только** если система может ответить:

1. почему именно это (разговор / дело / …);  
2. с кем или в каком контексте;  
3. почему именно сейчас (день / этап цели);  
4. что изменится, если пользователь это сделает;  
5. какой сигнал привёл к выводу;  
6. насколько вывод надёжен;  
7. что делать практически (наблюдаемое действие).

Нет ответов → текст **недействителен**.

## 1. Источники

Что реально использовано (фиксируется в trace), например:

- дата, место, время рождения и **точность времени**  
- эфемериды, координаты, часовой пояс  
- натальные положения, текущие транзиты  
- нумерологические входы  
- карта / число дня (после reveal)  
- ответы ритуала, check-in  
- история, поведенческие сигналы  
- текущие цели и активные обязательства  

Нет источника в списке → нельзя опираться на него в тексте.

---

## 2. Расчёт (код, не модель)

Детерминированные величины считает **код**. Модель их не угадывает.

### Пример: асцендент

Фиксировать в расчёте / limitations:

- локальное звёздное время  
- широта / долгота  
- эклиптическая долгота восточного горизонта  
- система домов  
- точность времени рождения  
- **невозможность** точного ASC при неизвестном времени → `partial` / omit, не «примерно Лев»

### Пример: число дня

- дата рождения + календарная дата  
- правило редукции  
- обработка мастер-чисел  
- **версия формулы**  
- промежуточные шаги  

Результат без `calculation_version` невалиден для production.

---

## 3. Смысловой вывод

Значение сигналов — по утверждённой системе правил, весов и конфликтов ([RFC DailyState K3](../rfc/RFC_DAILY_STATE_V0.md)), не свободная фантазия.

Примеры допустимых machine-claims:

- высокая вероятность импульсивной реакции  
- повышенная потребность в завершении  
- конфликт темпа дня и текущего ресурса  
- усиленная тема переговоров  
- низкая надёжность из-за неполных данных  

При слабых данных — **снижать конкретность**, не повышать уверенность.

---

## 4. Практическое применение

Через **ситуации**, не через меню сфер:

- ответить сразу или отложить решение  
- начать новое или закончить старое  
- тяжёлый разговор или сначала собрать факты  
- три задачи или одна  
- взять обязательство или сократить нагрузку  
- проявить инициативу или оставить пространство другому  

Паттерны поведения первичны; «работа/любовь» — только если следует из контекста пользователя ([EXPLAINABLE_INTERPRETATION.md](./EXPLAINABLE_INTERPRETATION.md)).

---

## 5. Пользовательский текст

Короткий грамотный вывод человеческим языком наставника ([TODAYFLOW_VOICE_CANON.md](../content/TODAYFLOW_VOICE_CANON.md)).

Формируется **только после** шагов 1–4.

## Цели: день · месяц · год

Цель нельзя предлагать потому, что она красиво звучит.

Учитывать: текущие цели, срок, этап, незавершённые действия, ресурс, историю выполнения, приоритет, контекст дня, ограничения.

| Плохо | Нормально |
|-------|-----------|
| Заверши одно важное дело. | До 15:00 отправь клиенту финальную версию предложения, которое осталось без ответа со вчера. Сегодня лучше закрывать уже начатые договорённости, чем открывать ещё одну задачу. |

**Правило честности:** если система не знает о конкретном предложении — **не придумывать**.  
Либо выбор из существующего списка задач пользователя, либо не показывать персональную цель.

### Иерархия (не четыре независимые генерации)

```text
год → месяц → неделя → день
```

**Годовая цель** раскладывается на:

- желаемый результат  
- исходное состояние  
- измеримый критерий  
- срок  
- ограничения  
- промежуточные этапы  
- ежемесячные действия  
- признаки отклонения  
- правило пересмотра  

**Месячная** связана с годовой или с явно выбранным направлением.  
**Дневная** — конкретный шаг месячной (или явно выбранной) цели.

## Обещания и аскезы

Поведенческий **контракт**, не магическое обещание и не опасный совет.

Система должна знать:

- чего хочет достичь пользователь  
- какое поведение мешает  
- что готов ограничить и на какой срок  
- как измеряется выполнение  
- какие исключения допустимы  
- не несёт ли ограничение вреда  
- что делать **вместо** запрещённого поведения  

| Плохо | Осмысленно |
|-------|------------|
| На 30 дней отказаться от удовольствий ради внутренней силы. | В течение 14 дней не открывать рабочие чаты после 20:30. Цель — уменьшить вечернее напряжение и восстановить сон. Исключение — заранее отмеченная экстренная ситуация. Каждый вечер отмечать соблюдение и как быстро получилось уснуть. |

---

## Что должно быть в каждом промпте

Промпт **не** просит «написать красивый прогноз».

Передаёт:

- проверенные факты  
- рассчитанные значения  
- степень надёжности  
- активные противоречия  
- пользовательский контекст  
- запрещённые выводы  
- формат результата  
- требование практической применимости  
- запрет придумывать отсутствующие данные  

### Обязательные правила для модели

1. Не добавлять фактов, которых нет во входе.  
2. Не выбирать сферу жизни без основания.  
3. Не выдавать общий совет за персональный.  
4. Не использовать туманные метафоры вместо действия.  
5. Не повторять смысл разными словами.  
6. Не давать категоричных прогнозов.  
7. Разрешать конфликты сигналов, а не скрывать их.  
8. При слабых данных снижать конкретность, а не повышать уверенность.  
9. Каждый совет связывать с конкретным наблюдаемым паттерном.  
10. Не писать текст, который нельзя применить в заявленном горизонте (сегодня / этот этап цели).

Эти 10 правил — **во всех разделах**, в каждом промпте генерации пользовательского текста.

## Обязательная трассируемость

Для каждого результата внутри системы сохраняются:

| Поле | Смысл |
|------|--------|
| `source_inputs` | что реально подано |
| `calculation_version` | версия формул / эфемерид / редукций |
| `ruleset_version` | версия словаря интерпретации |
| `prompt_version` | версия промпта |
| `model_version` | модель генерации (если была) |
| `evidence` | подтверждённые сигналы |
| `derived_claims` | структурированные выводы до текста |
| `confidence` | надёжность (или null) |
| `limitations` | что неизвестно / partial |
| `generated_output` | финальный текст |
| `created_at` | время |
| `fingerprint` | стабильный ключ воспроизводимости |

Внутренний вопрос, на который всегда есть ответ:

> Почему пользователю X в дату D был показан именно этот текст?

Ответ — по trace, не «это сгенерировала модель».

---

## Критерий готовности любого вывода

Перед production:

1. Есть источники?  
2. Расчёт детерминирован и версионирован (где применимо)?  
3. Интерпретация по ruleset, не из воздуха?  
4. Практика привязана к реальному контексту (или честно не персональна)?  
5. Текст проходит проверки [EXPLAINABLE_INTERPRETATION.md](./EXPLAINABLE_INTERPRETATION.md)?  
6. Trace сохранён?  
7. Команда может объяснить происхождение за 2 минуты?

Нет на любой пункт → **не в production**.
