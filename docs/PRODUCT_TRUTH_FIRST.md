# Product Truth First

**Статус:** ACTIVE — обязательный продуктовый принцип TodayFlow  
**Владелец:** Product + Engineering  
**Аудитория:** product, design, engineering, агенты

---

## Принцип

> **Сначала рабочая логика, данные и API. Потом интерфейс.**  
> Дизайн отображает построенный продукт, а не изображает будущий.

Цель: не урезать построенный продукт, а **честно и последовательно показать то, что уже работает**.

**Umbrella:** [EXPLAINABLE_COMPUTATION_AND_INTERPRETATION.md](./EXPLAINABLE_COMPUTATION_AND_INTERPRETATION.md) — любой вывод = источник → расчёт → интерпретация → практика → текст; иначе не production.

**Production App Shell (закрыт):** [PR2_APP_SHELL.md](./PR2_APP_SHELL.md) — единая инфраструктура UI (sidebar · main · optional data-rail · shared states). К shell не возвращаемся, кроме багфиксов. Запрет filler-rail / reserved empty column / per-section shells — канон.

**Следующий execution slice:** [PR3_TODAY_PRODUCTION_SURFACE.md](./PR3_TODAY_PRODUCTION_SURFACE.md) — сначала `day_story_v1` (evidence / confidence / limitations / claim map / trace), затем контракт независимых блоков Today, затем surface. Не с Figma.

Связь с [TODAYFLOW_PRODUCT_MODEL.md](./TODAYFLOW_PRODUCT_MODEL.md): Personal Model остаётся SoT персонализации; этот канон запрещает UI, который **притворяется**, будто модель/API уже отдали результат.

**Следующая архитектурная фаза дня:** [DAILY_INTERPRETATION_ENGINE_PHASE.md](./DAILY_INTERPRETATION_ENGINE_PHASE.md) — единый `DailyState` (эволюция DayContext/DayModel), а цвет/камень/аромат/практика только как производные Recommendation Engine. Не добавлять независимые генераторы символов дня.

**Следующий артефакт (не UI):** [rfc/RFC_DAILY_STATE_V0.md](./rfc/RFC_DAILY_STATE_V0.md) — слои evidence / interpretation / recommendations, provenance, разрешение конфликтов сигналов.

**Объяснимая интерпретация:** [EXPLAINABLE_INTERPRETATION.md](./EXPLAINABLE_INTERPRETATION.md) — ни одно практическое утверждение без цепочки обоснования; советы только из DailyState; четыре проверки текста (обоснованность · практичность · конкретность · авторский язык).

**Аудит / gate:** [audits/EXPLAINABLE_GENERATION_AUDIT_REGISTRY_V0.md](./audits/EXPLAINABLE_GENERATION_AUDIT_REGISTRY_V0.md) — реестр generators × violations × remediation (не новый принцип).

## Жёсткие правила

1. **Нет backend-источника — нет блока** в production UI.
2. **Нет реальных данных** — показываем честное состояние и конкретное действие, а не заполненную заглушку.
3. **Fallback не может выглядеть как персональный расчёт.** Он должен быть обозначен как общий вариант либо скрыт.
4. **Нельзя добавлять фиктивные** проценты, графики, теги, историю, статусы обновления и рекомендации.
5. **Новая функция проходит цепочку:** модель данных → вычисление/генерация → API-контракт → обработка состояний → UI.
6. **Макеты** строятся только из подтверждённого реестра возможностей, уже существующих в коде.
7. **Экспериментальные будущие элементы** могут находиться в документации или прототипе, но не в рабочем интерфейсе.

## Паспорт UI-блока (обязателен)

Для каждого production UI-блока до реализации фиксируется:

| Поле | Описание |
|------|----------|
| **Source** | Backend endpoint или локальный достоверный источник |
| **Contract field** | Поле контракта / модели |
| **Appear when** | Условия появления блока |
| **States** | `loading` / `ready` / `empty` / `partial` / `error` |
| **User action** | Действие пользователя |
| **No data** | Поведение при отсутствии данных |

Если паспорта нет — **блок не реализуется**.

### Четыре типа блока

Каждый блок относится ровно к одному типу:

| Тип | Смысл | Пример |
|-----|--------|--------|
| **A. Реальный результат** | Есть источник и понятное происхождение | карта дня, дата рождения, завершённые дни |
| **B. Действие** | Можно выполнить сейчас | начать практику, закрыть день |
| **C. Объяснение результата** | Поясняет уже показанный результат | почему рекомендована практика |
| **D. Честное пустое состояние** | Данных нет; один следующий шаг | «Заверши первую практику…» |

Всё остальное удаляется или скрывается.

## Реестр состояний данных

| Состояние | UI |
|-----------|-----|
| `loading` | Скелетон / спиннер без выдуманных цифр |
| `ready` | Реальные поля контракта |
| `empty` | Что отсутствует + почему + одна CTA |
| `partial` | Только подтверждённые поля; пробелы не маскировать |
| `error` | Понятная ошибка + «Повторить»; сохранить ввод пользователя |

**Запрещено** визуально смешивать `empty`, `error` и `ready` (типичный антипаттерн: `.catch(() => [])`).

---

## Антипаттерны

| Антипаттерн | Правильно |
|-------------|-----------|
| `current ?? catalog[0]` как «твоя практика дня» | `recommendationSource`; fallback помечен или скрыт |
| `energy ?? 50` в UI | Не показывать число без серверного значения |
| «обновлено сегодня» = время открытия страницы | Badge только при реальном `generated_at` |
| Дефолтные теги / нулевой график / пустая история как контент | Скрыть до появления данных |
| Точный % без измерения | Качественный уровень или скрыть |

## Порядок работ

1. Подтвердить источник и контракт в коде.  
2. Описать паспорт блока.  
3. Реализовать состояния `loading` / `ready` / `empty` / `partial` / `error`.  
4. Только затем — визуал и композиция.  
5. Макет без пункта реестра возможностей → прототип/docs, не production.

---

## Критерий готовности экрана

Для каждого блока можно ответить:

1. Из какого endpoint или модели берутся данные?  
2. Что показывается при отсутствии данных?  
3. Может ли fallback быть принят за персональный результат?  
4. Какое действие может выполнить пользователь?  
5. Нужен ли блок до появления истории?  
6. Не дублирует ли он данные на этом же экране?  
7. Использует ли общий компонент дизайн-системы?
